proyecto se acuerda con el usuario como ir entregando las versiones del
producto, y para fijar la fecha de entrega, el proyecto le avisa al equipo
de desarrollo y este cumple el proyecto. Es el elemento organizativo a

Producto: artefactos que se crean durante la vida del proyecto

proceso: conjunto de actividades necesarias para transformar los requisitos

Procesos afectan a las personas
Viabilidad del proyecto el proyecto no debe parecer imposible para que no 
le afecte a las personas

Gestion de riesgo: Cuando las personas sienten que los riesgos han sido
analizados y reducidos, se sienten incomodos

Planificacion del proyecto: Cuando las personas creen que la planificacion
de un proyecto no es realista, la moral se hunde

Sensacion de cumplimiento: En un ciclo de vida iterativo las personas 
reciben retroalimentacion frecuente, la cual lo lleva a conclusiones que le
permiten tomar decisiones.
Retroalimentacion frecuente + conclusiones aceleran el ritmo de trabajo

Convirtiendo recursos en trabajadores:
El recurso: Que personas van a trabajar en el equipo de desarrollo
Trabajador es el concepto de puesto, se lo define y lo asocia con algun 
flujo de trabajo, Ademas de que debe llevar adelante una actividad para 
crear un determinado artefacto.

Recurso se convierte en trabajadores porque la gente que ocupa el puesto 
requiere formacion y entrenamiento preciso seguidos de una cuidadosa
asignacion guiada por el analisis y una adecuada supervision

Roles son los papeles que cumple un trabajador, estos pueden asumir varios 
roles en relacion a otros trabajadores en los diferentes flujos de trabajo
fundamentales
Una persona puede ser muchos trabajadores en la vida de un proyecto

Artefacto se construye en cada puesto, asociado a un flujo

Flujo de trabajo requisito

Se enfoca en dos etapas, la primera busco la vision que lleva a determinar
que requisitos tiene mi sistema, una vez identificados, los veo como
casos de uso

Requisito candidato, se acepta y se les asocia un caso de uso

Las condiciones y capacidades que el sistema debe cumplir para que pueda
llegarse a un acuerdo entre el cliente, los usuarios y los desarrolladores
sobre lo que debe hacer el sistema

Captura de requisitos: Acto de descubrir lo que se debe construir en 
circunstancias dificiles. Esto ultimo porque no es una tarea sensilla, o
porque la informacion que le dan a uno no es completamente confiable

Enumerar los requisitos candidatos devuelve el artefacto Lista de 
caracteristicas
Comprender el contexto del sistema devuelve el artefacto Modelo de dominio 
y/o negocio
Capturar los requisitos funcionales devuelve el artefacto Modelo de casos 
de uso
Capturar los requisitos no funcionales. Casos de uso capturan estos ultimos
dos devuelve el artefacto Requisitos adicionales no funcionales

Casos de uso del negocio conozco los pasos del negocio en pequeños pasos
para tener pequeños problemas
Define a un alto grado de definicion, los pasos del negocio

Durante la vida del sistema, clientes, usuarios, desarrolladores tienen
ideas que podrian convertirse en requisitos, las ideas entran a una lista
de caracterisiticas que crece cuando se añaden nuevos elementos y decrece
cuando alguna se convierte en requisito y luego en uno o mas casos de uso
o cuando esta caracteristica no entra al proyecto

La lista de caracterisitcas se utiliza para la planificacion
Cada caracteristica tiene un nombre corto y una breve descripcion, 
definicion, informacion suficiente para poder hablar de ella en el proyecto

cada caracterisita tiene un conjunto de valores de planificacion que puede
incluir:
Estado(propuesto-incluido-validado)
Costo estimado de implementacion (recursos-horas personal)
Prioridad (critico-significativo-secundario)
Nivel de riesgo (critico-significativo-ordinario)

Para capturar los requisitos correctos y construir el sistema correcto, los
desarrolladores claves, el arquitecto, los analistas senior, requieren un
firme conocimiento del contexto en el que se emplaza el sistema

Las dos aproximaciones mas importantes para expresar el contexto del
sistema utilizado por los desarrolladores de software son:
Modelado del dominio con el diag de clases semantico. Describe los 
conceptos importantes del contexto como los objetos del dominio y enlaza
estos objetos unos con otros
Modelado del negocio consiste en describir los procesos existentes u 
observados con el objeto de comprenderlos

Clases se relacionan, objetos se enlazan. Enlaces son instancias de 
relaciones

Modelado del negocio consiste en describir los procesos existentes u
observables con el objeto de comprenderlos. Lleva a la construccion de los 
casos de uso del negocio

El modelo de dominio se concreta elaborando un diag de clases semantico que
captura los conceptos mas importantes del contexto del sistema

El modelo de negocio inicial se concreta elaborando casos de uso del
negocio y luiego definir que pasos del mismo forman parte del sistema a
desarrollar (Requisito) y para los cualse se formulan casos de uso
